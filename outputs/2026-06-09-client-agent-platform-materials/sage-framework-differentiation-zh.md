# Sage Agent 框架差异性说明

这份材料专门回答一个问题：

为什么 Sage 不只是“又一个 Agent 应用”，而是一套更适合企业落地的 Agent 框架？

## 1. 总结版

Sage 的差异不在于“也能接模型、也能接工具、也能做聊天”，而在于它对企业 Agent 的核心假设不同。

很多 Agent 框架默认 Agent 是一个“会调用工具的大模型会话”。Sage 默认 Agent 是一个“可被产品化、可被治理、可被复用、可被观察的任务执行单元”。

因此 Sage 的设计重点不是把模型包装成聊天，而是围绕真实任务交付，构建：

- 任务执行闭环
- 工具选择与权限边界
- Skills 业务资产化
- 多 Agent 协作
- 记忆与工作区连续性
- Workbench 可视化执行现场
- 多入口产品化部署

一句话：

Sage 把 Agent 从“模型调用框架”推进到“企业级任务执行框架”。

## 2. Sage 与常见 Agent 框架的根本差异

| 维度 | 常见 Agent 框架 | Sage |
| --- | --- | --- |
| 核心抽象 | LLM + tools + prompt | Agent as a governed task execution unit |
| 目标 | 让模型能调用工具 | 让 Agent 能稳定完成真实任务 |
| 工具处理 | 把工具列表暴露给模型 | 工具注册、推荐、运行时扩展、权限边界分层 |
| 业务知识 | 主要写进 prompt | 用 Skills 沉淀为可维护资产 |
| 长任务 | 依赖上下文窗口和简单循环 | 规划、执行、自检、压缩、记忆和工作区协同 |
| 多 Agent | 通常是 demo 式协作 | 主 Agent / 子 Agent / 子会话 / 结果汇总的任务分工 |
| 可见性 | 多数停留在日志 | Workbench 展示文件、工具结果、产物和执行过程 |
| 产品化 | 偏开发者 SDK | Web、Desktop、CLI、浏览器、IM、自动化任务等多入口 |
| 企业治理 | 往往后补 | 从工具边界、沙箱、权限、可观测性开始设计 |

## 3. 差异一：Sage 不是“工具调用循环”，而是“任务执行闭环”

许多 Agent 框架的基本循环是：

`用户问题 -> 模型思考 -> 工具调用 -> 工具结果 -> 模型回答`

这个模式适合 demo，也适合简单任务，但企业场景里经常不够。银行真正需要的是：

`目标理解 -> 任务拆解 -> 工具执行 -> 状态观察 -> 错误恢复 -> 自检 -> 交付 -> 留痕`

Sage 更强调后者。

### 对客户的表达

普通 Agent 框架解决的是“模型怎么调用工具”。Sage 解决的是“一个任务如何被 Agent 稳定推进到交付”。

### 为什么这对银行重要

银行任务往往不是一次问答：

- 客户经理要准备客户沟通，不只是问一个产品信息
- 客服要识别问题、查资料、生成话术、判断升级、总结工单
- 营销要生成触达策略、筛选客户、识别意向、交接线索
- 运营要处理文档、比对规则、生成报告、留下记录

这些都需要执行闭环，而不是一次性回答。

## 4. 差异二：Sage 的工具体系不是“接得多”，而是“选得准、扩得稳、管得住”

很多框架会强调能接很多工具。但企业落地时，工具多本身会带来新问题：

- 模型上下文被大量工具 schema 占满
- 工具太多导致模型选择不稳定
- 不该用的工具被误用
- 临时需要的工具没有进入上下文
- 权限和审计难以统一管理

Sage 的差异在于工具体系分层：

1. 全局工具注册：平台知道有哪些工具
2. Agent 级授权：某个 Agent 被允许使用哪些工具
3. 会话级工具边界：当前会话不能突破授权范围
4. Tool suggestion：为当前任务选择最相关工具
5. Runtime expansion：任务中途需要更多工具时再扩展
6. Workbench / 日志：工具调用过程可见

### 对客户的表达

Sage 不只是工具多，而是工具使用更像企业权限体系：先授权，再选择，必要时扩展，全程可见。

### 可以打的差异点

“很多框架把所有工具都塞给模型，让模型自己选。Sage 更像一个受控工具调度系统：Agent 只看到当前最需要的工具；需要新增工具时，可以申请扩展，但不能越过权限边界。”

## 5. 差异三：Sage 用 Skills 承载业务流程，而不是把业务都塞进 prompt

普通 Agent 项目很容易变成 prompt 工程：

- 业务规则写在 prompt 里
- 场景流程写在 prompt 里
- 输出格式写在 prompt 里
- 合规要求也写在 prompt 里

短期可演示，长期会失控。

Sage 把 Skills 作为一等能力。Skill 不只是提示词片段，而是一个可维护的业务能力包，可以包含：

- 业务说明
- 操作流程
- 模板
- 脚本
- 示例
- 资产文件
- 工具使用方式

### 对客户的表达

Sage 的 Skills 让银行的业务方法论变成平台资产，而不是藏在一个不可维护的超长 prompt 里。

### 为什么这对银行重要

银行业务变化快：

- 产品策略变
- 合规话术变
- 客户分层规则变
- 活动打法变
- 内部流程变

如果这些都写在 prompt 中，后期维护很困难。用 Skills 后，可以把业务规则模块化、版本化、复用化。

## 6. 差异四：Sage 的多 Agent 不是炫技，而是为了企业任务分工

多 Agent 很容易被讲成“多个机器人聊天”。这不是企业最关心的。

Sage 的多 Agent 更接近企业里的任务分工：

- 主 Agent 负责理解目标、组织任务、汇总结果
- 子 Agent 负责专业领域判断或具体执行
- 子任务可以独立运行在子会话中
- 结果回到主 Agent 形成统一交付

### 银行例子

客户经营任务可以拆成：

- 客户画像 Agent：整理客户背景
- 产品匹配 Agent：查找适配产品
- 合规检查 Agent：审查表达边界
- 触达运营 Agent：生成跟进计划
- 汇总 Agent：输出客户经理可用材料

### 对客户的表达

Sage 的多 Agent 不是让多个模型互相讨论，而是把银行复杂任务拆给不同专业角色，并由主 Agent 统一交付。

## 7. 差异五：Sage 把“工作区”作为 Agent 执行的一部分

很多 Agent 框架的中心是聊天记录。Sage 的中心不只是聊天，还有工作区。

工作区里可以有：

- 上传文件
- 生成文件
- 中间过程材料
- 报告
- 图表
- 代码
- 表格
- 音视频或其他多格式产物

这意味着 Agent 的工作不是只存在于对话中，而是可以形成可检查、可复用、可交接的产物。

### 对客户的表达

Sage 不是让 Agent 只在聊天窗口里说话，而是让 Agent 在一个可见工作区里做事。

### 为什么这对银行重要

银行内部协作依赖材料和记录：

- 客户简报
- 工单记录
- 活动复盘
- 合规检查结果
- 审批材料
- 报告和表格

这些不能只是一段聊天回答，必须成为可交付产物。

## 8. 差异六：Sage 更接近“Agent 产品平台”，不是单一开发框架

一些框架更适合开发者搭建 demo 或后台服务。Sage 同时考虑了 Agent 的产品入口：

- Web
- Desktop
- CLI
- Chrome extension
- IM
- 自动化任务
- Workbench
- 嵌入式体验

### 对客户的表达

Sage 的目标不是只让开发者能写一个 Agent，而是让企业能把 Agent 放进真实业务现场。

### 为什么这对银行重要

银行不同角色的工作入口不同：

- 高管看经营洞察
- 客户经理在 CRM
- 坐席在客服系统
- 运营在活动后台
- IT 在管理控制台
- 一线员工可能在 IM 或浏览器里工作

Agent 能力必须进入这些入口，而不是要求所有人去一个新的聊天页面。

## 9. 差异七：Sage 的稳定性来自架构组合，而不是单点技巧

Agent 稳定性不是靠一句“我们 prompt 写得好”解决的。

Sage 的稳定性来自多个层面的组合：

- 任务规划降低复杂任务的不确定性
- 工具调用把回答变成可执行动作
- Tool suggestion 降低工具选择噪音
- Runtime expansion 处理任务中途能力不足
- Skills 降低业务规则漂移
- Memory 和工作区支持长期任务连续性
- Self-check 提供结果检查
- Workbench 让过程可见、可复盘
- 沙箱和权限边界降低执行风险

### 对客户的表达

Sage 的稳定性不是来自某一个模型技巧，而是来自从任务、工具、知识、执行、检查、可见性到治理的一整套框架设计。

## 10. 更锋利的竞品对比口径

### 对比 1：如果对方是 Chatbot / RAG 平台

他们的强项：

- 知识问答
- 文档检索
- FAQ 自动化

Sage 的差异：

- RAG 可以成为 Sage 的一个工具或能力
- Sage 不止回答知识，还能推进流程
- Sage 能把知识检索连接到工具执行和业务交付

一句话：

RAG 告诉你答案，Sage 帮你把答案变成动作和产物。

### 对比 2：如果对方是通用 Agent SDK

他们的强项：

- 开发灵活
- 快速搭 demo
- 工具调用基础能力

Sage 的差异：

- 更强调企业任务闭环
- 有产品入口和 Workbench
- 有 Skills 作为业务资产层
- 有工具推荐和运行时扩展
- 有多 Agent 与工作区协同

一句话：

通用 SDK 帮开发者搭 Agent，Sage 帮企业运营 Agent。

### 对比 3：如果对方是单点行业 Agent

他们的强项：

- 单个场景可能做得深
- 上线快
- 业务包装强

Sage 的差异：

- 不绑定单一场景
- 可以沉淀工具和 Skills
- 可以复制到相邻业务线
- 可以逐步形成 Agent 工厂

一句话：

单点 Agent 解决一个业务问题，Sage 帮银行建立持续解决业务问题的能力。

### 对比 4：如果对方是 RPA / 工作流自动化

他们的强项：

- 固定流程执行稳定
- 系统集成明确
- 规则化任务效率高

Sage 的差异：

- 能理解自然语言目标
- 能处理非结构化信息
- 能动态选择路径
- 能结合模型推理和工具执行

一句话：

RPA 执行预先定义好的流程，Sage 处理需要理解、判断和交付的知识工作。

## 11. 给高管的框架级表达

可以这样讲：

我们认为 Agent 的竞争不只是模型能力竞争，而是 Agent 框架能力竞争。企业真正需要的是一套机制，把模型能力变成可部署、可治理、可复用、可观察的任务执行能力。Sage 的差异在于，它从一开始就围绕真实任务交付设计，而不是围绕聊天界面设计。工具、Skills、多 Agent、记忆、Workbench 和多入口部署，都是为了让 Agent 从 demo 进入生产业务。

## 12. 给 IT 团队的框架级表达

可以这样讲：

Sage 的框架核心是把 Agent 执行拆成几个可治理层：模型负责理解和推理，工具层负责连接外部能力，Skills 层负责业务流程资产化，Session / Memory 负责上下文连续性，Workbench 负责过程可见，Sandbox / 权限边界负责执行安全，多入口产品层负责把同一能力嵌入不同业务现场。这样做的结果是，Agent 不再是一个孤立 prompt，而是一个可运营的软件系统。
