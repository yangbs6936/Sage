# Sage Agent 平台沟通弹药库：面向 某某 银行高管与 IT 团队

适用场景：客户来中国调研 Agent 落地实践，希望理解我们如何基于 Sage 构建可落地、可扩展、可治理的银行 Agent 产品。

这份材料不是 PPT 页面，而是用于内部备讲、现场答疑、会后补充和方案深化的“弹药库”。PPT 可以从中抽取 6-10 页，销售话术可以从中抽取 1-2 分钟版本，技术交流可以从中展开 30-60 分钟版本。

## 1. 一句话定位

Sage 是一套企业级 Agent 平台，不只是聊天机器人或单点 Copilot。它帮助银行把 Agent 从“能回答问题”推进到“能理解任务、调用工具、执行流程、验证结果、交付产物，并在多个渠道中稳定运行”的阶段。

更短版本：

Sage 让银行可以从单个 Agent 场景出发，逐步沉淀成可复制、可治理、可扩展的 Agent 平台能力。

## 2. 为什么银行现在需要 Agent 平台，而不是更多单点 AI Demo

银行客户通常已经看过很多 AI Demo，但真正落地时会遇到几个共性问题：

- Demo 能回答问题，但不能稳定完成业务流程
- 单个场景能跑，但换一个业务线就需要重新做
- 工具和系统很多，但 Agent 不知道什么时候该用哪个
- 大模型能力强，但执行过程不可见、不可控、不可追责
- 业务规则、合规要求、产品策略经常变化，提示词维护成本很高
- 前台、客服、营销、运营、IT 各自做 Agent，能力无法复用

Sage 要解决的不是“有没有一个 AI 助手”，而是“银行如何建立 Agent 产品化能力”。

## 3. 核心主张：从 Chatbot 到 Agent Workforce

可以把银行的 Agent 演进分成四层：

| 阶段 | 典型形态 | 价值 | 局限 |
| --- | --- | --- | --- |
| Chatbot | 问答机器人 | 降低人工问答压力 | 只能回答，难以执行 |
| Copilot | 辅助员工生成内容和建议 | 提升客户经理、客服、运营效率 | 仍然依赖人工串联流程 |
| Agent | 可调用工具并完成任务 | 能执行具体业务流程 | 需要治理、工具、记忆、权限和可观测性 |
| Agent Workforce | 多 Agent 协同的业务运营能力 | 可跨场景、跨渠道、跨流程复制 | 需要平台化能力支撑 |

Sage 的定位在第三层和第四层之间：既能快速落地具体 Agent，又能把能力沉淀为企业平台。

## 4. Sage 的六个核心差异点

### 4.1 完整任务闭环：不是只回答，而是能交付

传统 AI 助手常见能力是“理解问题并生成答案”。Sage 更关注完整任务闭环：

- 理解目标：识别用户真正要完成的业务结果
- 规划步骤：把复杂任务拆成可执行动作
- 调用工具：连接知识库、系统、文件、浏览器、IM、搜索、图像、问卷等能力
- 观察反馈：读取工具结果和环境变化
- 自检修正：发现错误后修复或调整路径
- 交付结果：生成材料、更新系统、发出消息、形成记录

客户可理解类比：

普通 Chatbot 像一个“会回答问题的顾问”；Sage Agent 更像一个“能做事、能留痕、能交付的数字员工”。

### 4.2 工具丰富，但不是工具堆砌

银行业务里工具和系统非常多：CRM、核心系统、知识库、营销平台、IM、工单、文档、审批、报表、搜索、浏览器、内外部 API。Agent 真正落地时，关键不只是“接入很多工具”，而是：

- 当前任务到底需要哪些工具
- 不需要的工具是否会干扰模型判断
- 中途发现需要新工具时，能否扩展
- 扩展是否仍然受权限控制
- 每次工具调用是否可见、可检查

Sage 的设计思路：

- 统一工具层：把内置工具、MCP 服务、浏览器自动化、搜索、IM、图像生成、问卷流程等统一成 Agent 可调用能力
- Tool suggestion：根据当前任务选择最相关工具，减少上下文噪音
- Runtime tool expansion：任务中途需要更多能力时，Agent 可以申请扩展工具
- 会话级权限边界：工具扩展不能突破该 Agent 原本允许的能力范围

对客户的表达：

我们不是让 Agent 面对一个无限大的工具箱，而是让它先拿到当前任务最合适的工具；需要更多工具时，也只能在授权边界内扩展。这让 Agent 既聪明，又可控。

### 4.3 Skills：把业务经验沉淀成可维护资产

银行 Agent 失败的一个常见原因是：业务规则全部写在提示词里。这样短期可以演示，长期会很难维护。

Sage 使用 Skills 的方式，把业务流程、模板、脚本、资产和操作规范封装成可复用能力包。

Skills 可以承载：

- 产品推荐流程
- KYC / 客户画像分析流程
- 客户跟进话术与节奏
- 合规检查规则
- 投顾内容生成规范
- 营销活动复盘模板
- 客诉处理 SOP
- 文档处理和报告生成流程

价值：

- 业务规则变化时，可以升级 Skill，而不是重写整个 Agent
- 多个 Agent 可以复用同一个 Skill
- 不同业务线可以基于共用 Skill 做轻量定制
- 业务专家可以参与沉淀流程资产，而不只是依赖技术团队

对客户的表达：

Sage 不是把业务知识散落在一个很长的 prompt 里，而是把可复用流程沉淀为 Skills。这样银行越用，平台资产越厚。

### 4.4 多 Agent 编排：复杂任务可以分工协作

银行里的复杂任务很少只靠一个角色完成。例如一个客户经营任务，可能涉及：

- 客户经理助手：理解客户背景和关系历史
- 产品专家 Agent：匹配合适产品和权益
- 合规审查 Agent：检查表达是否合规
- 营销运营 Agent：生成触达计划和活动复盘
- 数据分析 Agent：统计响应和转化

Sage 支持通过主 Agent 与子 Agent 协作，把复杂任务拆给更专业的 Agent。

价值：

- 主 Agent 负责目标理解和总体推进
- 子 Agent 负责专业判断或具体执行
- 复杂任务可以并行或分阶段推进
- 结果汇总回主 Agent，形成统一交付

对客户的表达：

单 Agent 适合简单任务，多 Agent 适合银行里的复合流程。Sage 的优势是可以从一个助手扩展成一个 Agent 团队。

### 4.5 Workbench 与可观测性：让执行过程看得见

银行 IT 和风控最关心的问题之一是：Agent 到底做了什么？

Sage 的平台设计强调可见性：

- 工具调用可见
- 文件和中间产物可见
- 执行进度可见
- 生成物可检查
- 会话和上下文可追踪
- 出错时可以定位在哪一步

Workbench 的意义：

它不是一个装饰性的 UI，而是让人类团队能够检查 Agent 的工作现场。对于企业落地，这比单纯聊天窗口重要得多。

对客户的表达：

银行不能接受一个“黑盒 Agent”。Sage 的 Workbench 让 Agent 的工作过程变成可检查、可复盘、可运营的过程。

### 4.6 多入口部署：同一能力进入不同业务现场

银行场景天然分散：

- 客户经理在 CRM
- 客服在坐席系统
- 管理者看报表
- 运营在营销平台
- 内部员工使用 Web 或桌面工具
- 客户触达发生在 IM、电话或 App

Sage 支持多种产品入口和部署形态：

- Web
- Desktop
- CLI
- Browser extension
- IM integration
- Embedded surface
- Automation task

价值：

同一套 Agent 能力不必绑定在一个界面里，可以进入不同业务现场。

对客户的表达：

我们不是让银行员工换一个新系统工作，而是把 Agent 能力嵌入他们已经工作的地方。

## 6. 与常见竞品方案的差异

### 6.1 与普通 Chatbot 的差异

普通 Chatbot：

- 主要回答问题
- 依赖固定知识库
- 不擅长执行流程
- 出错后难以恢复
- 交付结果通常是文本

Sage Agent：

- 可以规划和执行任务
- 可以调用工具和系统
- 可以生成文件、报告、消息和系统动作
- 可以自检和修正
- 可以沉淀工作过程和产物

一句话：

Chatbot 解决“问答”，Sage 解决“完成任务”。

### 6.2 与单点 Copilot 的差异

单点 Copilot：

- 绑定一个页面或一个流程
- 场景复制成本高
- 业务逻辑常常散落在 prompt 中
- 难以形成跨部门平台资产

Sage：

- 可复用运行时
- 可复用工具层
- 可复用 Skills
- 可复用 Agent 配置
- 可进入多个业务入口

一句话：

Copilot 是一个功能，Sage 是构建多个 Agent 产品的平台。

### 6.3 与只做 RAG 的方案差异

RAG 方案：

- 重点是知识检索和回答
- 适合制度问答、产品问答、资料查询
- 对执行任务、调用系统、交付产物支持有限

Sage：

- RAG 可以成为其中一个能力
- 同时支持工具调用、工作流、文件操作、IM 触达、多 Agent 和自动化
- 可以把知识问答变成业务动作

一句话：

RAG 让 Agent 知道答案，Sage 让 Agent 能把答案转化为行动。

### 6.4 与流程机器人 RPA 的差异

RPA：

- 擅长固定流程自动化
- 规则明确时效率高
- 对开放式任务和自然语言变化适应弱

Sage：

- 擅长开放式任务理解
- 可以动态规划步骤
- 可以结合工具、知识和多 Agent
- 能处理更多非结构化输入

一句话：

RPA 自动化固定动作，Sage Agent 自动化带判断的知识工作。

## 7. 客户高管可能关心的问题与回答口径

### Q1：这和我们已有的聊天机器人有什么不同？

回答：

已有聊天机器人通常解决“客户问什么，系统答什么”。Sage 更关注“客户或员工想完成什么任务”。它不仅能回答，还能规划步骤、调用工具、生成材料、触发流程、交付结果，并让过程可见。这是从问答系统到任务执行系统的差异。

### Q2：为什么不能每个场景单独做一个 Agent？

回答：

单独做当然可以快速启动，但长期会产生重复建设：工具接入重复、权限治理重复、业务规则重复、监控和运维重复。Sage 的价值是把底层运行时、工具、Skills、记忆、Workbench 和部署入口统一起来，让每个新场景不是从零开始。

### Q3：Agent 会不会失控？

回答：

Agent 是否可控，取决于平台设计。Sage 不鼓励让模型随意调用所有能力，而是通过 Agent 配置、会话级工具白名单、工具推荐、运行时扩展边界、沙箱和 Workbench，把能力限制在授权范围内，并让执行过程可检查。

### Q4：如何保证稳定性？

回答：

稳定性来自几个层面：任务拆解降低复杂度，工具调用让 Agent 操作真实系统，自检机制帮助发现错误，Workbench 让过程可观察，Skills 减少提示词漂移，多入口共享服务架构降低重复实现。对于生产场景，我们会从低风险、高价值流程开始，逐步扩大范围。

### Q5：如何处理银行合规要求？

回答：

合规要求不应该只靠模型“自觉遵守”。更好的方式是把合规规则沉淀为 Skills、审查步骤、工具权限和人工确认节点。对于高风险动作，可以要求人工确认；对于内容生成，可以引入合规审查 Agent 或规则检查流程。

### Q6：如果内部系统很多，Agent 怎么接？

回答：

Sage 的统一工具层可以接入内置工具、MCP 服务和银行内部 API。不同系统可以被封装为工具，Agent 不需要理解每个系统的底层实现，只需要在任务中调用合适工具。这样既能扩展系统能力，也能保持统一治理。

### Q7：Sage 是否绑定某个模型？

回答：

平台的核心价值不是绑定某个模型，而是围绕 Agent 执行构建运行时、工具、Skills、记忆和可观测性。模型能力会持续变化，但企业需要一套能吸收模型进步、同时保持业务治理的平台。

### Q8：如何从试点走向规模化？

回答：

建议采用三步：

1. 选一个高价值、低风险、边界清晰的场景试点
2. 把试点中的工具、流程和规则沉淀为 Skills 和平台能力
3. 复用这些能力扩展到相邻场景，逐步形成 Agent 产品矩阵

## 9. 建议现场讲法

### 90 秒版本

我们认为银行 Agent 的核心不只是做一个聊天机器人，而是建立一套能持续复制的 Agent 平台。Sage 的特点是，它让 Agent 可以完成完整任务闭环：理解目标、规划步骤、调用工具、验证结果、交付产物。它还有统一工具层，可以接入银行内部系统、知识库、IM、浏览器和自动化工具；同时通过工具推荐和运行时扩展，让 Agent 在能力丰富的情况下仍然保持可控。业务规则则通过 Skills 沉淀为可维护资产，而不是散落在 prompt 里。最终，银行可以从一个客户经理助手或客服助手开始，逐步扩展到营销、运营、IT 等更多场景，形成自己的 Agent 工厂。

### 3 分钟版本

今天很多 Agent 项目卡在两个地方：第一，Demo 可以回答问题，但不能稳定完成流程；第二，单个场景可以做，但无法复制到更多业务线。Sage 解决的是平台化问题。

从能力上看，Sage 支持规划、执行、自检和交付。Agent 可以调用工具、检索知识、操作文件、触发流程、发送消息，并把产物留在 Workbench 中供团队检查。工具很多时，Sage 不会把所有工具都塞给模型，而是通过 tool suggestion 选择当前任务相关工具；如果中途需要更多工具，可以 runtime expansion，但仍然限制在权限边界内。

从业务复用上看，Sage 用 Skills 承载业务流程、模板、脚本和规则。比如 KYC、产品推荐、客户跟进、合规检查、活动复盘，都可以沉淀为可维护资产。这样新场景不是从零开始，而是在已有平台能力上组装。

对 某某 来说，我们建议从高价值、低风险场景开始，例如客户经理 Copilot、呼叫中心坐席助手、营销线索筛选或内部文档运营。成功后，把工具和 Skills 复用到相邻场景，逐步形成一个银行级 Agent 平台。

### 8 分钟版本结构

1. 先承认客户看到过很多 Agent Demo
2. 指出真正落地的挑战不是模型，而是平台化
3. 用 Sage 的任务闭环解释“能回答”和“能交付”的差别
4. 用工具推荐和扩展解释“能力丰富但可控”
5. 用 Skills 解释“业务规则可复用、可维护”
6. 用 Workbench 解释“过程可见、可复盘”
7. 用 CMB 场景连接 某某 的前台、客服、营销、运营机会
8. 给出从试点到平台的路线图

## 10. 从试点到平台的落地路线图

### Phase 1：场景验证

目标：

- 选择一个边界清晰的高价值场景
- 验证 Agent 是否能提升效率或转化
- 保留人工确认和人工接管

推荐场景：

- 客户经理 Copilot
- 呼叫中心坐席助手
- 营销线索初筛
- 内部文档处理

交付物：

- 场景 Agent 原型
- 工具接入清单
- 业务流程 Skill 初版
- 效果评估指标

### Phase 2：能力沉淀

目标：

- 将试点中重复出现的流程沉淀为 Skills
- 将系统接入沉淀为工具
- 建立权限、日志、人工确认和复盘机制

交付物：

- Skill library
- Tool registry
- Agent 配置模板
- Workbench 运营规范

### Phase 3：多场景复制

目标：

- 扩展到相邻业务场景
- 支持多个 Agent 或子 Agent 协同
- 建立可运营的 Agent 产品矩阵

交付物：

- 多业务线 Agent
- 共享工具和 Skills
- 运营看板
- 场景复用方法论

### Phase 4：Agent 工厂

目标：

- 让银行内部具备持续构建、治理和迭代 Agent 的能力
- 不依赖每个场景从零开始外包开发

交付物：

- Agent 创建和配置流程
- 统一治理框架
- 内部开发者和业务专家协作机制
- 长期运营和评估体系

## 11. 可放进 PPT 的强表达

以下句子适合直接放页脚、章节页或高管摘要：

- Agent 的价值不在于多会聊天，而在于能否稳定完成业务任务。
- 银行需要的不是更多 AI Demo，而是一套可复制的 Agent 平台能力。
- Sage 让每个成功试点都沉淀为下一批场景的能力资产。
- 工具越多，越需要选择机制、权限边界和过程可见性。
- Skills 让业务规则成为可维护资产，而不是不可控的提示词堆叠。
- Workbench 让 Agent 从黑盒执行变成可检查、可复盘的工作过程。
- 从 Copilot 到 Agent Workforce，关键差异是能否规划、执行、验证和交付。
- Sage 的平台优势来自工具、Skills、记忆、编排和多入口部署的组合效应。

## 12. 可用于会后邮件的总结段落

基于本次交流，我们建议将 某某 的 Agent 建设从单点场景试点升级为平台化路径。Sage 可以作为底层 Agent 平台，支持任务规划、工具调用、Skills 场景复用、多 Agent 编排、记忆、可视化 Workbench 和多入口部署。短期可以从客户经理辅助、呼叫中心坐席助手、营销线索筛选或内部运营文档处理等高价值场景切入；中期将工具、流程和业务规则沉淀为可复用资产；长期形成面向零售银行、财富管理、客服、营销、运营和 IT 的 Agent 产品矩阵。

## 13. 内部注意事项

- 不要把 Sage 讲成单一模型能力，重点讲平台能力
- 不要只说“我们有很多工具”，要讲工具选择、工具扩展和权限边界
- 不要把 Skills 讲成普通 prompt，要讲成可维护的业务流程资产
- 不要过度承诺完全自动化，银行场景要强调 human-in-the-loop
- 不要只讲前台场景，也要讲客服、运营、IT 的平台化价值
- 不要攻击竞品，强调“单场景 Agent”和“平台型 Agent”的差异即可
- 对高管讲业务复制和增长，对 IT 讲治理、接入、可见性、稳定性
